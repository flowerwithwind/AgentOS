## 行为约束

涉及前端 + 后端 + 数据库中两层以上的改动时，必须输出影响范围矩阵（层 → 变更文件 → 影响模块 → 风险）。修改已有 API 的请求/响应格式时，必须标注兼容性：`✅ 向前兼容`（仅新增字段）、`⚠️ 破坏性变更`（修改或删除已有字段）、`🔁 可灰度兼容`（新旧格式并存）。对外接口新增字段必须保持向后兼容——只做增量不做减量，老字段保持不动，新字段给默认值或 null。数据库 Schema 变更（DDL）必须同时提供正向 SQL 和回滚 SQL，两者成对出现，回滚 SQL 注释掉防止误执行。改字段名这种危险操作建议拆成"新增字段 → 双写数据 → 废弃老字段 → 删除老字段"四步走。生成代码时优先遵循项目已有的代码风格，先扫一眼同模块文件的缩进、命名、注释习惯再动手。

## MCP 工具调用约定

| 场景 | 调用方式 |
|------|----------|
| 实现完成并本地验证通过后 | `github/create_pull_request({"owner": "...", "repo": "...", "title": "feat(xxx): ...", "head": "feature/xxx", "base": "develop", "body": "变更内容...兼容性...数据库变更..."})` |
| 需要搜索某个库的参考写法 | `github/search_code({"query": "关键词"})` |
| 开发验证时查看页面渲染效果 | `playwright/browser_navigate({"url": "http://localhost:10086/..."})` |
| 开发验证时检查 JS 报错 | `playwright/browser_console_messages({"url": "..."})` |
| 开发验证时拦截 API 请求检查响应 | `playwright/browser_network_requests({"url": "..."})` |

## Skill 调用约定

| Skill | 触发时机 |
|-------|----------|
| `frontend-scaffold` | 需要新建页面或前端模块时，一次性生成模板 + CSS + JS + 路由注册代码 |
| `backend-scaffold` | 需要新建 API 接口或数据模型时，按路由 → Handler → Repository → DB 顺序生成全链路代码，DB 层必须附带回滚 SQL |

## 工作流

P0（必须先确认）：扫目录 → 读现有同类模块代码 → 理解风格  
P1（核心开发）：调用 `frontend-scaffold` skill 生成前端 → 调用 `backend-scaffold` skill 生成后端（附回滚 SQL）→ 联调  
P2（验证）：用 `playwright/browser_navigate` 看页面 → `browser_console_messages` 查报错 → `browser_network_requests` 确认 API 链路  
P3（提交）：输出影响范围矩阵 → 调用 `github/create_pull_request` 提交 PR，描述中写清楚变更内容、兼容性、数据库迁移脚本及回滚方案

## 原则

1. 改一处要看三层——前端改了模板就检查后端返回字段是否匹配，数据库查询是否返回了足够的字段
2. 接口兼容性不是选项——对外暴露的接口只做增不做减，新增字段给默认值
3. 数据库迁移要有退路——每条 DDL 都要配回滚 SQL，注释掉放在旁边
4. 脚手架代码不等于完成——生成的模板代码要手动过一遍异常路径和边界情况
5. 开发完必须提交 PR——调 `github/create_pull_request` 走评审流程，PR 描述说清楚改了啥、影响谁、怎么回滚
