## 第一章 项目背景与现状

### 1.1 项目定位

XHAgentOS 是我们团队在西华大学课程实训期间开发的一套 AI Agent 管理系统。说白了，最初的想法是做一个能帮企业做舆情监控和智能问答的平台——用户可以在上面配置 AI 模型、管理数字员工、做信息采集，还能直接跟 AI 对话问数据相关的问题。整个系统围绕"智能瞭望"（也就是舆情采集分析）和"智能问数"（对话式数据查询）两个核心场景展开。

选这个方向其实有个很现实的原因：市面上类似的产品要么功能单一（只做舆情或者只做对话），要么价格太高，我们希望做一个集成度高、上手门槛低的系统，算是对"AI 赋能数据分析"这个命题的一次实践。

### 1.2 技术栈概览

技术选型上我们用的是 Tornado 6.4 作为 Web 框架。选 Tornado 主要考虑它原生支持异步（async），虽然项目里目前大部分接口还是同步写法，但留了异步扩展的空间。数据库这块做了 SQLite 和 MySQL 的双引擎支持——开发阶段用 SQLite 零配置很方便，部署时再切到 MySQL，中间通过一个 SQL 翻译层（`app/models/db.py` 里的 `_MySQLConnection` 类）来抹平两种数据库的语法差异。

前端用的 Layui 2.13.7，这是一个比较传统的 CSS+JS 框架，选择它主要是因为团队成员相对熟悉，而且它自带的表格、表单组件能省不少前端工作量。AI 调用用的 OpenAI 官方 Python SDK，但 base_url 可以自定义，所以也能对接 DeepSeek 等兼容 OpenAI 协议的国内模型。

完整的技术栈清单：Tornado 6.4（异步 Web 框架）、SQLite + MySQL（双数据库引擎）、Layui 2.13.7（前端 UI 框架）、OpenAI SDK（兼容 DeepSeek）、jieba（中文分词，词云用）、fpdf2（PDF 导出）、beautifulsoup4（网页解析采集用）、apscheduler（定时任务，目前声明了但还没完全用起来）、pymysql（MySQL 驱动）。

### 1.3 用户角色

系统的角色体系比较简单，目前内置了两个角色：超级管理员（role_id=1）和普通用户（role_id=2）。超级管理员拥有所有功能权限，普通用户只能用智能问数和后台首页。同时支持自定义角色的创建，理论上可以扩展出更细的权限分配，但实际上权限粒度只做到了菜单级别——也就是说只能控制某个角色能不能看到某个菜单页面，没法控制页面内的具体操作（比如能不能删除某条记录）。

角色和权限的绑定关系存在 `role_permissions` 表里，是 role_id 和 function_id 的多对多关系。种子数据（`app.py` 的 `seed_data()` 函数）会在系统初始化时自动创建两个默认角色，并给超级管理员分配全部 22 个功能菜单。

### 1.4 功能模块总览与成熟度评估

系统目前一共有 16 个业务模块，这里逐个说说每个模块做到什么程度了。

**认证模块（基本完成）**：包括登录、注册、退出三个功能，鉴权用 Tornado 自带的 `set_secure_cookie` + `@authenticated` 装饰器实现。密码加密用了 PBKDF2-HMAC-SHA256 加随机 salt（见 `app/models/user.py`），这个方案安全性还行，比单纯的 MD5 强很多。不过 cookie 那边有些问题后面会讲。

**用户管理（完成）**：CRUD 用户、分配角色，功能都能用。但缺少用户状态管理（没有启用/禁用），也没有操作审计记录。

**功能/角色/权限（完成）**：树形菜单管理加角色权限绑定。功能表支持 parent_id 做层级，配合 sort_order 排序。权限管理页面做了角色-功能的二级联动选择器。问题是权限粒度太粗，只是菜单级。

**模型引擎（较完善）**：支持多模型管理（增删改查、批量操作、设默认），Token 用量追踪（按次记录，有日趋势图），还有在线对话测试。不过只支持 OpenAI 协议，没有 Ollama 本地模型或者 Anthropic 之类的接入。

**瞭望管理（较完善）**：这块功能链比较长——信息源管理（百度新闻、搜狗新闻两个默认源）→ 关键词采集（通过 URL 模板 + BeautifulSoup 解析）→ 数据仓库（存储采集结果，支持筛选去重）→ 深度采集（用 crawl4ai 抓取完整网页内容）。整体流程跑得通，但采集源只有两个，也没做定时自动采集。

**数字员工管理（完成）**：CRUD、启用/禁用、设置系统提示词、关联技能，基本功能都有。但数字员工目前本质上也只是一个"带 system prompt 的对话角色"，缺少记忆能力（不记得之前的对话上下文以外的东西）。

**技能管理（框架性完成）**：这个模块说实话做得比较空——skills 表只有 name 和 description 两个业务字段，技能本质上就是个文本标签，没有任何可执行逻辑。跟数字员工的关联也仅仅是在 system prompt 里拼上技能描述文本。

**对话系统-用户侧（较完善）**：支持多会话管理、@数字员工切换角色、选择不同模型、PDF 导出对话记录。功能层面该有的都有，但 AI 回复是非流式的，用户得等整个回复生成完才能看到内容，体验上有点拖。

**对话管理-管理侧（完成）**：管理员可以查看、筛选、导出（CSV）所有用户的对话记录。缺质量评估机制。

**会话管理（完成）**：管理用户会话（session），查看会话内的消息详情，支持删除。基本功能到位。

**API Token 管理（完成）**：可以创建和删除 API Token，Token 有 call_count 字段。但说实话这个模块是个半成品——Token 没有实际的鉴权中间件来验证请求，call_count 也不会自增，等于说 Token 创建了但没地方真正用它。

**数智大屏（较完善）**：用了 ECharts GL 做 3D 地球可视化，还有词云（jieba 分词）、采集趋势图、模型 Token 使用排行。视觉效果不错，但数据维度比较单一，基本就是数数（count）然后展示。

**系统设置（较完善）**：站点名称、主题色、Logo 上传、默认模型配置、采集参数、日志查看和清理。功能比较齐全，但 Logo 上传没有做文件类型和大小校验，这个安全隐患后面细说。

**数据库切换（完成）**：SQLite 和 MySQL 双引擎在线切换，有 SQL 翻译层做语法兼容，还支持连接测试。但切换时不做数据迁移，等于说从 SQLite 切到 MySQL 后数据是空的。

**智能问数（半成品）**：目前管理侧的智能问数只是一个 iframe 直接嵌了用户侧的对话页面（`AdminQAHandler` 渲染 `user_chat.html`），没有任何针对数据查询场景的特殊处理。

## 第二章 现状评估（六维度评分）

为了比较客观地评估项目当前的状态，我从六个维度分别打了 1-5 分。说实话分数不太好看，但我觉得认清问题比假装一切都好更重要。

| 维度 | 评分 | 现状描述 | 核心问题 |
|------|------|---------|---------|
| 代码质量与工程化 | 2.5 | 核心功能可运行，但代码组织存在明显问题 | admin.py 单文件 1663 行含 80+ Handler 类（God Object）；无配置管理体系；测试覆盖率极低 |
| 安全性 | 2.0 | 基础认证框架存在但漏洞较多 | 5 个 Handler 缺少认证装饰器；cookie_secret 硬编码；API Key 明文；无 RBAC 中间件 |
| 性能 | 2.5 | 小规模数据下可用，存在多个性能隐患 | 数字员工列表 N+1 查询；AI 调用同步阻塞；无缓存机制；无数据库连接池 |
| 前端与用户体验 | 3.0 | 页面功能完整，但工程化程度低 | CSS/JS 全内联在模板中；base.html 未被所有页面复用；无前端构建工具 |
| 可维护性与可扩展性 | 2.5 | 模块边界基本清晰（model/controller 分层），但内部耦合严重 | Controller 层包含业务逻辑；无中间件层；无数据库迁移工具；新增路由需改 3 处文件 |
| DevOps 与部署 | 1.5 | 几乎没有任何 DevOps 基础设施 | 无 Dockerfile；无 CI/CD 流水线；无环境变量管理；无健康检查端点 |

### 2.1 代码质量与工程化（2.5 分）

最突出的问题就是 `app/controllers/admin.py` 这个文件——1663 行代码里塞了 80 多个 Handler 类，从用户管理到模型引擎到瞭望采集到系统设置全挤在一起。每次改一个模块的功能都得在这个大文件里翻来翻去找对应的类，效率很低，merge conflict 的概率也高。这其实是典型的 God Object 反模式（一个类或文件承担了过多的职责）。

工程化方面，整个项目没有统一的配置管理方案。像 `cookie_secret`、数据库路径、日志级别这些配置项，要么硬编码在 `app.py` 里，要么分散在各个文件中。`debug=True` 也是写死的（`app.py` L75-76），生产环境如果忘了改就是个大问题。

数据库迁移用的是 try/except 模式——`app.py` 的 `migrate_db()` 函数里用 try/except 包裹 ALTER TABLE，如果列已存在就忽略。这种方式能用但很脆弱，没法追踪迁移历史和回滚。

### 2.2 安全性（2.0 分）

安全问题是我觉得最需要紧急处理的。先列一下发现的具体问题：

**未认证接口**：`app/controllers/admin.py` 中 SystemSetting 系列的 5 个 Handler 类缺少 `@tornado.web.authenticated` 装饰器。具体来说，`SystemSettingHandler`（L1273）、`SystemSettingApiHandler`（L1290）、`SystemSettingLogoUploadHandler`（L1348）、`SystemSettingLogApiHandler`（L1373）、`SystemSettingLogClearHandler`（L1400）这些类的 `get()` 或 `post()` 方法都没有认证保护。这意味着任何未登录用户都可以直接访问系统设置、修改配置、上传文件甚至清空日志。另外 `DigitalEmployeeByNameApiHandler`（L1193）也没有认证装饰器，虽然这个可能是故意给外部调用的，但应该有别的鉴权机制。

**硬编码密钥**：`app.py` L72 的 `cookie_secret` 被硬编码为 `"demo-cookie-secrete-change-me"`。如果生产环境不改这个值，攻击者就可以伪造任意用户的 Cookie 实现越权登录。

**API Key 明文**：`app.py` L301 在种子数据里直接写了一个 DeepSeek 的 API Key（`sk-25369530ebd443cfa5cddfde5b819b2b`）。虽然这是示例数据，但 Key 存在数据库里也是明文，models 表的 `api_key` 字段没有做任何加密处理。

**默认管理员密码**：`app.py` L287 创建默认管理员时密码是 `admin123`，这在任何部署场景下都是一个安全隐患。

**Logo 上传无校验**：`SystemSettingLogoUploadHandler`（L1348）在处理文件上传时，没有对文件类型做白名单校验，也没有限制文件大小。攻击者理论上可以上传任意文件（比如 .py 脚本）到服务器的 static 目录。

**Cookie 安全属性缺失**：`set_secure_cookie()` 调用时没有显式设置 `httponly=True` 和 `secure=True`，Cookie 可能被 XSS 攻击窃取。

### 2.3 性能（2.5 分）

**N+1 查询问题**：`app/models/digital_employee.py` 的 `list()` 方法（L97-108）在获取数字员工列表时，先查出所有员工记录，然后在 for 循环里逐个查询每个员工关联的技能名称。如果有 20 个数字员工，就会产生 1+20=21 次数据库查询。同样的问题也出现在 `get_by_id()` 方法里。正确做法是用一条 JOIN 查询一次获取所有数据。

**AI 调用同步阻塞**：`app/controllers/user_chat.py` 的 `_handle_send()` 方法（L154-168）中，OpenAI SDK 的 `client.chat.completions.create()` 是同步调用。Tornado 是单线程事件循环模型，一个同步的 HTTP 请求（可能耗时 5-30 秒）会阻塞整个事件循环，导致其他所有用户的请求都被卡住。

**词云全量加载**：`WordCloudApiHandler`（admin.py L1446）在做词云数据计算时，会把符合条件的所有瞭望记录和对话记录都加载到内存，然后用 jieba 分词处理。数据量大时内存和时间开销都很可观。

**无缓存无连接池**：每次 `get_connection()` 调用都会新建一个数据库连接（SQLite 或 MySQL），没有连接池复用。数据库配置也是每次从文件读取。Dashboard 和 BigScreen 的统计数据查询也没有任何缓存机制。

### 2.4 前端与用户体验（3.0 分）

前端给了 3 分主要是因为功能层面确实做得比较完整，该有的页面都有。但代码组织上问题不少：大部分页面的 CSS 和 JavaScript 都是直接内联在 HTML 模板里的，没有抽取成独立文件。这导致了很多重复代码——比如 `getCookie()` 函数在好几个模板里都有定义。

`base.html` 作为基础模板，并没有被所有页面继承使用。有些页面（比如登录页）是完全独立的 HTML，而管理后台的页面又是另一套布局（`admin.html`），两套体系之间没有统一的样式规范。

没有使用任何前端构建工具（Webpack/Vite 之类的），也没有模块化的概念。JavaScript 代码散落在各个模板的 `<script>` 标签里，无法做 tree-shaking 和压缩。

### 2.5 可维护性与可扩展性（2.5 分）

项目的分层设计（controllers / models / templates）本身是合理的，Model 层用 Repository 模式做了数据访问的封装。但 Controller 层里混入了太多业务逻辑——比如 `ModelChatHandler` 里直接 new 了一个 OpenAI 客户端做 API 调用，这种代码应该抽到 Service 层。

没有中间件层的设计。认证是通过 Tornado 的装饰器实现的，但像权限校验、请求日志、异常处理这些横切关注点（cross-cutting concerns）都是各个 Handler 自己处理的，没有统一的拦截机制。

新增一个路由需要改 3 个地方：对应的 Controller 文件、`app/__init__.py` 的 import 列表、`app.py` 的路由注册表。这个过程容易遗漏，也增加了维护成本。

### 2.6 DevOps 与部署（1.5 分）

这个维度分数最低。项目目前没有任何容器化支持（没有 Dockerfile），没有 CI/CD 流水线（没有 GitHub Actions 或类似配置），没有环境变量管理机制（所有配置都是硬编码或写死在文件里的），也没有健康检查端点（`/health` 之类的）。日志输出是纯文本格式写到文件里，没有结构化，也不支持日志轮转——时间长了 app.log 会越来越大。

## 第三章 横向优化需求

这一章按优先级把需要做的优化项梳理清楚。优先级划分的原则是：安全和稳定性相关的最高优先，架构和性能改进次之，工程化和体验改善放最后。

### 高优先级（安全/稳定性相关）

**H1 - 修复未认证接口**

问题描述：SystemSetting 系列 5 个 Handler 类缺少 `@tornado.web.authenticated` 装饰器，导致未登录用户可以访问系统设置、上传文件、查看和清空日志。

影响范围：`app/controllers/admin.py` L1273-L1410 涉及的 SystemSettingHandler、SystemSettingApiHandler、SystemSettingLogoUploadHandler、SystemSettingLogApiHandler、SystemSettingLogClearHandler。

优化方案：给上述 5 个 Handler 的所有 HTTP 方法加上 `@tornado.web.authenticated` 装饰器。同时审计一遍所有其他 Handler，确保没有遗漏。

验收标准：未登录状态访问 `/admin/settings`、`/admin/settings/api`、`/admin/settings/logo` 等 URL 时，自动跳转到登录页面。

**H2 - 移除硬编码密钥**

问题描述：cookie_secret（`app.py` L72）、示例 API Key（`app.py` L301）、默认管理员密码（`app.py` L287）全部硬编码在源码中。

影响范围：`app.py` 全局配置和种子数据。

优化方案：引入环境变量机制。cookie_secret 从 `os.environ.get("COOKIE_SECRET")` 读取，提供开发默认值但在生产环境强制要求配置。API Key 不写在种子数据里，改为首次启动时提示用户通过管理界面配置。默认管理员密码改为首次登录强制修改。

验收标准：源码中不包含任何真实的密钥或密码字符串；通过环境变量可以正常配置所有敏感信息。

**H3 - 实现 RBAC 权限中间件**

问题描述：当前权限只做到菜单级别，无法控制具体操作的权限。且权限校验散落在各个 Handler 中，没有统一机制。

影响范围：`app/controllers/base.py` 的 BaseHandler 以及所有继承它的 Handler。

优化方案：在 BaseHandler 中增加 `prepare()` 方法（Tornado 的请求预处理钩子），根据当前请求的 URL 和用户角色做权限校验。同时扩展权限模型，支持操作级权限（如"用户可以查看但不能删除"）。

验收标准：普通用户访问管理类接口时返回 403；权限校验逻辑集中在 BaseHandler 中，单个 Handler 不需要手动写权限判断代码。

**H4 - 文件上传安全加固**

问题描述：Logo 上传接口没有文件类型白名单和大小限制。

影响范围：`app/controllers/admin.py` L1348-L1370 的 SystemSettingLogoUploadHandler。

优化方案：增加文件类型白名单（仅允许 png/jpg/gif/webp），限制文件大小不超过 2MB，对上传的文件名做安全处理（去除路径穿越字符）。

验收标准：上传 .py、.exe 等非图片文件时返回错误提示；上传超过 2MB 的文件时返回错误提示。

**H5 - 数据库密码保护**

问题描述：切换到 MySQL 时，数据库密码会以明文存入 `system_settings` 表和 `database/db_config.json` 文件。

影响范围：`app/models/db.py` 的 `switch_database()` 函数和 `app/controllers/admin.py` 的 DbSwitchApiHandler。

优化方案：数据库密码仅存储在 `db_config.json` 文件中（不写入数据库表），并在 `DbSwitchApiHandler` 的 GET 接口中确保密码字段不返回给前端（当前已有 `config.pop("password", None)` 的处理，这点做得不错）。进一步可以对配置文件中的密码做简单加密或掩码处理。

验收标准：API 响应中不包含数据库密码明文；`system_settings` 表中不再存储 `db_password` 字段。

**H6 - Cookie 安全加固**

问题描述：`set_secure_cookie()` 调用时未设置 `httponly` 和 `secure` 属性。

影响范围：`app/controllers/auth.py` L22 的 LoginHandler。

优化方案：登录时调用 `set_secure_cookie("username", username, httponly=True)`，并根据环境配置 `secure=True`（仅 HTTPS 环境）。同时在 `BaseHandler.get_current_user()` 中增加 Cookie 完整性校验。

验收标准：浏览器 DevTools 中查看 Cookie 属性，httponly 标志为 true；在 HTTPS 部署环境下 secure 标志为 true。

### 中优先级（架构/性能相关）

**M1 - 拆分 admin.py**

问题描述：admin.py 单文件 1663 行，包含 60+ Handler 类，是典型的 God Object。

优化方案：按业务模块拆分为独立文件——`admin_user.py`（用户管理）、`admin_model.py`（模型引擎）、`admin_lookout.py`（瞭望管理）、`admin_system.py`（系统设置/数据库切换/日志）、`admin_digital.py`（数字员工/技能）、`admin_screen.py`（大屏/词云）、`admin_session.py`（会话/对话管理）。每个文件控制在 200-300 行。

验收标准：拆分后每个文件不超过 400 行；所有路由注册正常；功能回归测试通过。

**M2 - 修复 N+1 查询**

问题描述：数字员工列表 `DigitalEmployeeRepository.list()` 在循环内逐个查询技能关联。

优化方案：使用 LEFT JOIN 一条 SQL 获取所有员工及其技能关联，或者用 GROUP_CONCAT 聚合函数在数据库层面拼接技能名称。

验收标准：数字员工列表接口只执行 1-2 次数据库查询（通过日志或 SQL 监控验证）。

**M3 - AI 调用异步化**

问题描述：OpenAI SDK 的同步调用阻塞 Tornado 事件循环。

优化方案：用 `asyncio.to_thread()` 或 `tornado.ioloop.IOLoop.run_in_executor()` 将同步的 OpenAI 调用放到线程池执行，避免阻塞事件循环。长期考虑切换到 `openai.AsyncOpenAI` 客户端。

验收标准：AI 对话请求处理期间，其他用户的请求不受阻塞（可并发测试验证）。

**M4 - 引入数据库迁移工具**

问题描述：当前用 try/except 做数据库迁移，无法追踪历史和回滚。

优化方案：引入 Alembic 或自己实现一个轻量的版本化迁移方案（类似 Django 的 migrations），每个迁移脚本带版本号，按顺序执行。

验收标准：迁移历史可查询；支持向前迁移和回滚；迁移脚本有明确的命名规范。

**M5 - 增加数据库连接池**

问题描述：每次 `get_connection()` 都新建连接，没有复用机制。

优化方案：SQLite 场景下维护一个线程级连接池（或者复用同一个连接）；MySQL 场景下使用 `pymysql` 的连接池或引入 `DBUtils`/`SQLAlchemy.pool`。

验收标准：高并发场景下数据库连接数稳定在池大小范围内，无连接泄漏。

**M6 - 抽取公共 CSS**

问题描述：大量 CSS 内联在 HTML 模板中，重复代码多。

优化方案：将通用样式抽取到 `app/static/css/` 目录下的独立 CSS 文件中。目前已有 `base.css`、`design-tokens.css` 等文件，需要进一步整合各模板中的内联样式。

验收标准：HTML 模板中不再包含超过 20 行的内联 CSS；公共样式修改只需改一个文件。

**M7 - 引入配置管理**

问题描述：配置项散落在代码各处，无统一管理。

优化方案：使用 YAML 或 TOML 格式的配置文件（如 `config.yaml`），支持环境变量覆盖。区分 development / production / testing 三套配置。

验收标准：所有可配置项集中在一个配置文件中；通过环境变量 `XHAGENT_ENV=production` 可以切换配置集。

### 低优先级（工程化/体验改善）

**L1 - 补充测试**

为 Controller 层编写集成测试，覆盖核心接口（登录、用户管理、模型管理、对话等）的正常流程和异常处理。目标覆盖率 > 60%。

**L2 - 前端 JS 模块化**

将各模板中散落的 JavaScript 代码抽取为独立 JS 文件（如 `admin_common.js`、`chat.js`），公共函数（`getCookie()`、AJAX 封装等）统一管理。

**L3 - 日志系统升级**

从纯文本日志升级为结构化 JSON 日志格式，引入日志轮转（`RotatingFileHandler`），支持按日期和大小切割。

**L4 - Docker 支持**

编写 Dockerfile 和 docker-compose.yml，包含 Python 运行时、依赖安装、SQLite 数据卷挂载等。让项目能一键容器化部署。

**L5 - 健康检查端点**

新增 `/health` 接口，返回系统状态（数据库连接、模型配置、内存使用等），供运维监控系统使用。

**L6 - 修复死代码**

清理项目中未使用的 import、注释掉的代码块，以及 `scheduled_tasks` 表中声明但未使用的字段。修复 `ScheduledTaskRepository` 参数不匹配的问题。

## 第四章 纵向拓展需求

横向优化解决的是现有系统的质量和稳定性问题，纵向拓展则是在功能维度上做增量。这里我用了一个"影响力-复杂度"矩阵来分类，帮助判断哪些功能值得优先投入。

### 第一象限：高影响力 + 低难度（优先实施）

**D1 - 对话流式响应**

功能定义：将当前 AI 对话的非流式响应改为 SSE（Server-Sent Events）流式输出，让用户可以逐字看到 AI 回复，而不是等全部生成完才显示。

价值分析：这是提升用户体验最直接的一个改动。目前用户发一条消息可能要等 5-15 秒才能看到完整回复，流式输出可以把"感知等待时间"降到 1 秒以内。

实现复杂度：中等偏低。OpenAI SDK 本身支持 stream=True 参数，主要工作量在 Tornado 端实现 SSE 响应和前端的流式渲染。

与现有代码的关系：主要改造 `app/controllers/user_chat.py` 的 `ChatApiHandler`，以及 `user_chat.html` 模板的前端逻辑。

**E1 - 舆情分析系统**

功能定义：基于瞭望管理模块已采集的数据，利用 LLM 做情感分析、风险等级标注和舆情趋势预测。

价值分析：这是让瞭望模块从"数据采集工具"升级为"智能分析平台"的关键功能。目前采集了大量数据但只做了简单的展示和词云统计，分析深度不够。

实现复杂度：中等。LLM 驱动的分析主要靠 Prompt Engineering，不需要训练模型。核心是设计好分析维度和输出格式。

与现有代码的关系：复用 `lookout_records` 表的数据，新增分析结果存储和分析任务调度逻辑。

**D4 - 采集源扩展与定时调度**

功能定义：增加 RSS 订阅源作为采集渠道，激活已在 requirements.txt 中声明的 apscheduler 实现定时自动采集。

价值分析：目前只有百度新闻和搜狗新闻两个采集源，覆盖面太窄。RSS 是信息源最标准的分发格式，接入成本低但能覆盖大量新闻站点。定时采集则让系统从"手动触发"变成"自动运转"。

实现复杂度：低。RSS 解析有成熟的库（feedparser），apscheduler 已经在依赖列表里了。

与现有代码的关系：扩展 `app/models/lookout.py` 的采集器类，新增 RSS 采集器。在 `app.py` 启动时初始化调度器。

**E3 - 数据导出中心**

功能定义：整合现有的 CSV 导出（对话管理）和 PDF 导出（对话记录），新增 Excel 格式支持，提供统一的数据导出管理页面。

价值分析：数据导出是企业用户的高频需求，目前导出功能分散在不同模块里且格式有限。统一导出中心能提升专业感。

实现复杂度：低。openpyxl（Excel）已在依赖列表中，主要工作是 UI 整合。

与现有代码的关系：抽取 `ConversationRepository.export_csv()` 和 `ChatExportHandler._build_pdf()` 的导出逻辑到统一服务层。

**N5 - 提示词管理模块**

功能定义：为系统提示词提供版本管理、模板库和效果对比功能。管理员可以创建、编辑、回滚提示词模板，数字员工配置时从模板库中选择。

价值分析：目前数字员工的 system prompt 是一个纯文本框，修改后无法回滚，也没有模板复用机制。对于需要反复调优提示词的场景，版本管理非常必要。

实现复杂度：低。核心是 CRUD + 版本记录，数据模型简单。

与现有代码的关系：新增 `prompts` 表和对应的 Repository/Handler，修改数字员工管理模块关联提示词模板。

**E5 - 采集数据标注**

功能定义：利用 LLM 对瞭望采集到的数据自动分类打标（如行业类别、情感倾向、关联实体等）。

价值分析：自动标注能大幅减少人工整理数据的工作量，也为后续的舆情分析提供结构化数据基础。

实现复杂度：低-中。主要是 Prompt 设计和批量调用 LLM 的任务调度。

与现有代码的关系：扩展 `lookout_records` 表增加标签字段，新增标注任务处理器。

### 第二象限：高影响力 + 高难度（战略投资）

**D2 - 技能系统实质化**

功能定义：将当前的"文本标签"式技能升级为真正的可执行能力单元。每个技能可以关联一个 Function Calling 定义（OpenAI 的函数调用能力），数字员工在对话中可以触发技能执行实际任务（如查数据库、调 API、生成报表等）。

价值分析：这是让数字员工从"聊天机器人"进化为"智能 Agent"的核心升级。没有可执行的技能，数字员工本质上只是一个带 system prompt 的对话接口。

实现复杂度：高。需要实现 Function Calling 的完整链路——技能定义（JSON Schema）→ LLM 识别调用意图 → 参数解析 → 函数执行 → 结果回注对话。

与现有代码的关系：重构 `skills` 表结构（增加 function_schema、handler_code 等字段），改造 `user_chat.py` 的对话处理逻辑以支持 function calling 循环。

**E4 - 知识库管理系统（RAG）**

功能定义：实现文档上传 → 文本分段 → Embedding 向量化 → 向量数据库存储 → 语义检索的完整 RAG（Retrieval-Augmented Generation，检索增强生成）链路。用户可以将企业文档导入系统，对话时自动检索相关知识注入上下文。

价值分析：RAG 是当前解决 LLM 幻觉问题和领域知识不足的主流方案，也是企业级 AI 系统的标配功能。

实现复杂度：高。涉及文档解析（PDF/Word/Markdown）、文本分段策略、Embedding 模型调用、向量数据库（ChromaDB 或 FAISS）集成、检索排序等多个技术环节。

与现有代码的关系：新增 `knowledge_base` 相关的数据表和模块，修改对话处理链路以支持检索结果注入。

**N2 - 工具/插件市场**

功能定义：支持注册外部 API 工具（如天气查询、汇率换算等），并在对话中通过工具调用执行。类似 OpenAI 的 GPTs 插件机制。

价值分析：让系统从封闭走向开放，用户可以通过工具扩展 AI 的能力边界。

实现复杂度：高。需要工具注册框架、工具调用执行引擎、安全沙箱等。

与现有代码的关系：与技能系统（D2）有重叠，可以合并设计。

**S1 - RAG 知识增强**

功能定义：将知识库检索的结果自动注入到对话上下文中，让 AI 基于检索到的真实文档回答问题。

价值分析：这是 E4 的下游功能，两者配合才能形成完整的 RAG 方案。

实现复杂度：中。核心是 prompt 拼接和检索结果排序。

与现有代码的关系：修改 `_handle_send()` 中的消息构建逻辑，在 system prompt 和用户消息之间插入检索到的文档片段。

**N1 - 工作流编排引擎**

功能定义：提供可视化的 DAG（有向无环图）编辑器，让用户可以通过拖拽节点的方式编排多步骤任务流程（如"先采集 → 再分析 → 再通知"），并由执行引擎自动调度运行。

价值分析：让复杂的业务逻辑不再依赖代码开发，降低使用门槛。

实现复杂度：很高。需要前端 DAG 编辑器、后端执行引擎、任务状态管理、失败重试等一系列组件。

与现有代码的关系：是一个相对独立的新模块，但需要能调用现有的瞭望采集、AI 对话等能力作为流程节点。

### 第三象限：适时推进

**D5 - 权限精细化**：从菜单级权限扩展到操作级（增删改查分别控制）和数据级（用户只能看到自己创建的数据）。

**N3 - 监控告警中心**：搭建模型健康度、采集任务状态、系统资源使用率的仪表盘，设置阈值告警机制。

**N6 - 审计日志系统**：记录所有管理操作的审计日志（谁在什么时间做了什么），支持查询和导出。

**I1 - 开放 API 平台**：激活现有 API Token 模块的实际鉴权功能，提供 API 接口文档（Swagger/OpenAPI），让外部系统可以通过 API 调用本系统的能力。

**D3 - 模型引擎增强**：支持 Ollama 本地部署的模型、实现智能路由（根据问题类型自动选择最合适的模型）。

### 第四象限：远期规划

**N4 - 多租户与团队协作**：支持多个团队/组织共享同一套系统，数据隔离，协作管理。

**S2 - 多 Agent 协作编排**：多个数字员工之间可以互相通信和协作，共同完成复杂任务。

**I3 - 企业 IM 集成**：将对话能力接入钉钉、飞书等企业即时通讯工具。

## 第五章 核心功能需求详述

这一章从前面梳理出来的需求里挑了 8 个优先级最高的，逐个展开说清楚每个功能到底是什么、怎么验收、技术上要注意什么。选需求的标准综合考虑了用户价值、安全紧迫性和技术可行性。

### F01 对话流式响应（对应 D1）

**功能定义**：将当前 AI 对话的"等全部生成完才显示"改为逐字流式输出（SSE，Server-Sent Events），用户在发送消息后几乎立刻就能看到 AI 开始回复，类似 ChatGPT 的打字机效果。这个改动解决的核心问题是用户感知等待时间过长。

**用户故事**：As a 普通用户，I want to 在发送消息后立即看到 AI 逐字回复，so that 我不需要面对空白页面干等十几秒。

**功能描述**：用户在前端点击发送后，后端调用 OpenAI SDK 时使用 `stream=True` 参数。Tornado 端将响应类型设为 `text/event-stream`，把 SDK 返回的每个 chunk 以 SSE 格式（`data: {...}\n\n`）实时推送到前端。前端用 `EventSource` 或 `fetch` + `ReadableStream` 接收流式数据，每收到一个 chunk 就追加到消息气泡中。对话结束后发送 `data: [DONE]` 标记。流式传输过程中仍需记录 Token 用量（从最后一个 chunk 的 usage 字段提取）。

**验收标准**：
- 发送消息后 1 秒内开始显示第一个字
- 流式过程中消息内容平滑追加，无闪烁或重复
- Token 用量记录与非流式模式一致
- 网络中断时前端能优雅降级（显示已收到的部分内容）

**技术要点**：`app/controllers/user_chat.py` 的 `ChatApiHandler` 的 `_handle_send()` 方法（L113-L185）需要拆分为两个路径——普通模式和流式模式。Tornado 的 SSE 实现需要禁用请求缓冲（`self.flush()`）。前端 `user_chat.html` 需要改造消息渲染逻辑，支持 Markdown 增量渲染（可以考虑用 marked.js 的流式模式）。

**优先级**：高 | **复杂度**：中偏低

### F02 舆情分析系统（对应 E1）

**功能定义**：基于瞭望管理模块已采集的新闻数据，利用 LLM 进行自动化的情感分析（正面/中性/负面）、风险等级标注（高/中/低）和主题归类，生成结构化的舆情分析报告。

**用户故事**：As a 管理员，I want to 对采集到的新闻数据自动做情感分析和风险评级，so that 我能快速发现负面舆情和潜在风险，而不是人工逐条阅读。

**功能描述**：在数据仓库（`lookout_records` 表）中新增 `sentiment`（情感倾向）、`risk_level`（风险等级）、`topic_tags`（主题标签）三个字段。管理员可以在数据仓库页面触发"AI 分析"按钮，系统批量调用 LLM 对每条记录做分析，将结果回写到数据库。同时新增一个"舆情分析"看板页面，展示情感分布饼图、风险趋势折线图、高频主题排行等可视化数据。LLM 的 Prompt 需要设计好输出格式（JSON），确保返回结果可解析。

**验收标准**：
- 对选中的采集记录触发分析后，每条记录都有情感倾向和风险等级标注
- 分析结果在数据仓库列表中可查看、可筛选
- 舆情看板正确展示情感分布和风险趋势
- LLM 调用失败时单条记录标记为"分析失败"，不影响其他记录

**技术要点**：扩展 `app/models/db.py` 中 `lookout_records` 表的 schema（注意这里涉及数据库迁移，参考 M4 的迁移方案）。分析任务需要异步执行（类似深度采集 `DeepCollectTask` 的模式），避免阻塞请求。LLM 调用复用现有的 `ModelRepository.get_default()` 获取模型配置。

**优先级**：高 | **复杂度**：中

### F03 技能系统实质化（对应 D2）

**功能定义**：将当前仅有 name + description 两个字段的"文本标签"式技能，升级为可执行的能力单元。每个技能包含一个 Function Calling 的 JSON Schema 定义和对应的执行逻辑，数字员工在对话中可以由 LLM 自主决定调用哪个技能、传什么参数，系统执行后将结果回注对话。

**用户故事**：As a 管理员，I want to 给数字员工配置可执行的技能（如"查询数据库"、"搜索新闻"），so that 数字员工在对话中能真正帮用户完成任务，而不只是聊天。

**功能描述**：重构 `skills` 表，新增 `function_schema`（JSON 格式的函数定义，符合 OpenAI Function Calling 规范）和 `handler_type`（执行方式：内置函数/API 调用/SQL 查询）字段。技能管理页面增加 Schema 编辑器和测试面板。对话处理链路（`user_chat.py` 的 `_handle_send()`）需要改造为支持 Function Calling 的循环——发送消息时带上可用技能列表（`tools` 参数），如果 LLM 返回 `tool_calls`，系统解析并执行对应函数，将结果作为 `tool` 角色的消息回传，直到 LLM 生成最终文本回复。

**验收标准**：
- 技能管理页面可以定义包含参数 Schema 的函数描述
- 数字员工关联技能后，对话中 LLM 能根据用户意图自动调用对应技能
- 技能执行结果正确回注对话上下文
- Function Calling 循环不超过 5 轮（防止死循环）

**技术要点**：`app/controllers/user_chat.py` 的 `_handle_send()` 方法（L113-L185）是改造核心。OpenAI SDK 的 `chat.completions.create()` 调用需要传入 `tools` 参数，响应处理需要增加 `tool_calls` 分支。技能执行引擎可以用策略模式——不同 `handler_type` 对应不同的执行器类。`app/models/skill.py` 需要大幅扩展。

**优先级**：高 | **复杂度**：高

### F04 知识库管理与 RAG（对应 E4 + S1）

**功能定义**：实现完整的 RAG（Retrieval-Augmented Generation，检索增强生成）链路——管理员上传文档（PDF/Word/Markdown），系统自动分段、生成 Embedding 向量并存入向量数据库。用户对话时，系统先对问题进行向量检索，找到最相关的文档片段注入上下文，让 AI 基于真实资料回答问题。

**用户故事**：As a 管理员，I want to 上传企业文档建立知识库，so that 数字员工在对话中能引用文档内容回答专业问题，减少 AI 的胡编乱造。

**功能描述**：知识库管理模块包含文档上传（支持 PDF、DOCX、Markdown）、文档预览、分段策略配置（按段落/按固定长度/按语义）。向量化流程使用 Embedding API（可以复用已配置的模型，或者单独配置 Embedding 模型），存储到 ChromaDB（轻量级向量数据库，pip install chromadb）。对话时新增一个检索步骤——在调用 LLM 之前，先对用户消息做向量检索，取 Top-K 相关片段拼接到 system prompt 中。前端需要展示"引用来源"标记，让用户知道回答是基于哪些文档。

**验收标准**：
- 上传文档后能在 30 秒内完成分段和向量化（10 页以内文档）
- 对话提问时，系统能检索出相关文档片段并注入上下文
- AI 回答中标注引用来源（文档名 + 段落位置）
- 知识库支持增删改查，删除文档时同步清理向量数据

**技术要点**：这是整个 PRD 中技术复杂度最高的需求。涉及文档解析（PyPDF2/python-docx）、文本分段（需要设计合理的 chunk 策略，推荐 500-1000 字一段、overlap 100-200 字）、Embedding 生成、ChromaDB 集成、检索结果排序和 Prompt 拼接。建议分两期实现：一期做文档管理和基础 RAG，二期做高级检索（混合检索、重排序）。

**优先级**：高 | **复杂度**：很高

### F05 RBAC 权限中间件（对应优化 H3）

**功能定义**：在 `BaseHandler` 中实现统一的权限校验中间件，拦截每个请求并根据用户角色和请求 URL 判断是否有访问权限。从当前的"菜单级"权限扩展到"操作级"权限。

**用户故事**：As a 超级管理员，I want to 能控制不同角色对同一模块的不同操作权限（如只读/可编辑/可删除），so that 系统权限分配更灵活安全。

**功能描述**：扩展 `functions` 表，新增 `action` 字段（view/create/edit/delete），使权限粒度从页面级细化到操作级。在 `BaseHandler`（`app/controllers/base.py`）中重写 `prepare()` 方法，每次请求到达 Handler 之前先查角色权限表。权限数据在用户登录后缓存到 Cookie 或 Session 中，避免每次请求都查库。对于 API 类接口，如果权限不足返回 403 JSON 响应；对于页面请求，跳转到无权限提示页。

**验收标准**：
- 普通用户访问管理类 API 时返回 HTTP 403
- 权限配置变更后实时生效（不需要重新登录）
- 超级管理员不受权限限制
- 权限校验逻辑仅存在于 BaseHandler 中，单个 Handler 无需手写权限代码

**技术要点**：`app/controllers/base.py` 的 `BaseHandler` 新增 `prepare()` 钩子。需要建立一个 URL → 功能 ID → 所需操作的映射表（可以在路由注册时自动生成）。权限数据缓存策略需要设计好——太短影响性能，太长导致变更不及时生效。

**优先级**：高 | **复杂度**：中

### F06 安全加固套件（对应优化 H1 + H2 + H4 + H6）

**功能定义**：打包解决当前系统中所有高优先级安全问题，包括修复未认证接口、移除硬编码密钥、加固文件上传和 Cookie 安全属性。这个需求不涉及新功能开发，纯粹是安全漏洞修补。

**用户故事**：As a 系统运维人员，I want to 系统没有已知的安全漏洞，so that 生产环境部署时不用担心被攻击。

**功能描述**：具体包含四项修复工作。(1) 给 `admin.py` L1273-L1410 的 5 个 SystemSetting Handler 加上 `@authenticated` 装饰器。(2) 将 `app.py` L72 的 `cookie_secret` 改为从环境变量 `XHAGENT_COOKIE_SECRET` 读取，开发环境提供默认值，生产环境强制配置。(3) Logo 上传接口（`admin.py` L1348-L1370）增加文件类型白名单（png/jpg/gif/webp）和 2MB 大小限制。(4) 登录时 `set_secure_cookie()` 增加 `httponly=True` 参数。

**验收标准**：
- 未登录访问 `/admin/settings/api` 返回 302 跳转到登录页
- 源码中 `cookie_secret` 的值来自 `os.environ.get()`
- 上传 .exe 文件返回错误提示
- 上传 3MB 的 PNG 文件返回"文件过大"提示
- 浏览器 DevTools 中 Cookie 的 HttpOnly 属性为 true

**技术要点**：改动范围小但涉及面广，需要仔细回归测试确保不影响正常功能。建议编写自动化测试覆盖每个修复项。

**优先级**：高（最高） | **复杂度**：低

### F07 监控告警中心（对应 N3）

**功能定义**：提供系统运行状态的实时监控仪表盘，包括模型调用成功率/延迟、采集任务状态、数据库连接状态、系统资源使用率等指标，支持设置阈值告警（邮件或站内通知）。

**用户故事**：As a 管理员，I want to 在一个页面上看到系统各组件的运行状态，so that 出问题的时候我能第一时间发现和定位。

**功能描述**：新增"监控中心"页面，展示以下维度的数据——模型健康度（API 调用成功率、平均响应时间、Token 消耗趋势）、采集健康度（采集成功率、采集源可用性、最近一次采集时间）、系统健康度（数据库连接状态、磁盘空间、内存使用）。后台需要新增一个指标采集服务，定时记录各维度的指标数据（可以存到新建的 `system_metrics` 表中）。告警规则引擎做简单版——当某个指标超过阈值时生成告警记录，管理员在通知中心查看。

**验收标准**：
- 监控仪表盘展示最近 24 小时的各维度指标
- 模型 API 连续 3 次调用失败时生成告警
- 采集任务超过设定时长未完成时生成告警
- 告警记录可在管理后台查看和处理

**技术要点**：指标采集可以用 apscheduler 定时执行（正好把已经在 requirements.txt 里声明但没用起来的 apscheduler 用上）。前端仪表盘用 ECharts 实现（项目中数智大屏已有 ECharts 的使用经验）。

**优先级**：中 | **复杂度**：中偏高

### F08 工作流编排引擎（对应 N1）

**功能定义**：提供可视化的工作流编辑器，用户通过拖拽节点和连线的方式编排多步骤任务流程（如"定时采集新闻 → AI 分析情感 → 生成报告 → 发送通知"），由后端执行引擎按计划自动调度运行。

**用户故事**：As a 管理员，I want to 通过拖拽的方式创建自动化任务流程，so that 不需要写代码就能把多个功能串联起来实现复杂的业务逻辑。

**功能描述**：前端使用 DAG（有向无环图）编辑器（可以考虑 jsPlumb 或 LogicFlow 之类的开源库），支持以下类型的节点——触发器（定时/手动/Webhook）、瞭望采集节点、AI 分析节点、数据导出节点、通知节点。后端新增工作流引擎，负责解析 DAG、按拓扑顺序执行节点、管理节点间的数据传递。每个工作流有独立的运行日志，支持手动触发和暂停/恢复。工作流定义存储在 `workflows` 表中，运行记录存储在 `workflow_runs` 表中。

**验收标准**：
- 用户可以在画布上拖拽创建至少包含 3 个节点的工作流
- 工作流保存后可以手动触发执行
- 执行过程中每个节点的输入输出可查看
- 节点执行失败时工作流暂停，记录错误信息
- 支持定时触发器（cron 表达式）

**技术要点**：这是功能面最广的需求，前端和后端都有大量工作。前端 DAG 编辑器需要处理节点拖拽、连线、参数配置等交互。后端执行引擎需要实现 DAG 拓扑排序、节点调度、状态机管理。建议先做一个最小可用版本（只支持串行流程 + 手动触发），再逐步增加并行和定时触发。

**优先级**：中 | **复杂度**：很高

## 第六章 非功能性需求

功能性需求说完了"做什么"，非功能性需求说的是"做到什么程度"。这部分虽然不像功能那样直观，但对系统的实际可用性同样关键。

### 6.1 安全需求

所有 API 接口必须经过身份认证（`@authenticated` 或等效的中间件拦截），不能有漏网之鱼。敏感配置项（cookie_secret、数据库密码、API Key）不允许以明文形式出现在源码中，统一通过环境变量或加密配置文件管理。文件上传功能必须校验文件类型（白名单机制：png、jpg、gif、webp、pdf、docx、md）和大小限制（单文件不超过 2MB）。Cookie 必须设置 `httponly=True`，在生产环境部署时额外设置 `secure=True`。密码存储继续使用 PBKDF2-HMAC-SHA256 + 随机 salt 的方案（当前 `app/models/user.py` 的实现已经做到了这一点），迭代次数不低于 100,000 次。

### 6.2 性能需求

非 AI 接口的 API 响应时间（从收到请求到返回响应）应控制在 500ms 以内（P95）。AI 流式响应的首 token 延迟不超过 2 秒。系统应支持至少 50 个用户同时在线使用（对话、管理等操作），数据库层面能承受这个量级的并发查询。词云、大屏等统计类接口的响应时间可以放宽到 3 秒以内，但需要做缓存（至少 60 秒的缓存周期），避免每次刷新都全量计算。

### 6.3 可用性需求

系统整体可用性目标为 99.5%（全年停机不超过 44 小时，含计划内维护）。AI 模型调用失败时应优雅降级——显示错误提示而非页面崩溃，支持自动重试（最多 1 次）。采集任务失败时记录错误日志但不影响其他任务的执行。提供 `/health` 健康检查端点，返回系统各组件的状态摘要。

### 6.4 可维护性需求

单个源码文件不超过 500 行（当前 admin.py 有 1663 行，必须拆分）。Controller 层的核心业务逻辑需要有集成测试覆盖，目标覆盖率 > 60%。日志输出采用结构化 JSON 格式（包含时间戳、日志级别、模块名、消息内容），支持日志文件按日期轮转。代码提交需要有有意义的 commit message，分支管理遵循 Git Flow 或类似的规范。

### 6.5 兼容性需求

前端兼容 Chrome、Edge、Firefox 的最新两个主版本。管理后台的 Layui 框架不做移动端适配（桌面端优先），但用户侧的智能问数页面需要做基础的响应式布局，确保在手机浏览器上能正常使用对话功能。PDF 导出功能依赖系统中文字体（Windows 下用 simhei.ttf，Linux 下用 wqy-microhei），需要在文档中注明字体依赖。

## 第八章 风险与依赖

做任何项目规划都得想想可能踩的坑，这里把主要的风险点列出来。

### 8.1 技术栈制约

前端用的 Layui 是一个比较老的框架，社区活跃度在下降，现成的组件和插件越来越少。如果后续要做 DAG 编辑器、Markdown 流式渲染这些高级前端功能，Layui 本身帮不上什么忙，需要引入额外的 JS 库（LogicFlow、marked.js 等），这会导致前端技术栈变得混杂。另一个选择是逐步迁移到 Vue 3 或 React，但这个工作量太大了，短期内不现实。

Tornado 的异步模型也是个双刃剑。它的事件循环是单线程的，虽然支持 async/await，但项目里很多同步代码（比如 OpenAI SDK 调用、jieba 分词）如果在事件循环里直接执行就会阻塞。把同步代码改成异步或者放到线程池执行需要额外的工作量，改不好的话还容易引入并发 bug。

### 8.2 安全风险

前面第二章列的那些安全问题（硬编码密钥、未认证接口、明文密码等）如果不及时修复，系统一旦部署到公网环境就面临被攻击的风险。特别是 `cookie_secret` 硬编码这个问题——如果多个部署实例用了同一个 secret，攻击者可以跨实例伪造 Cookie。建议把 F06（安全加固套件）作为第一个迭代周期的任务，不跟其他功能混在一起，集中火力解决。

### 8.3 数据迁移风险

SQLite 和 MySQL 之间的切换目前只做了配置层面的切换和 SQL 语法翻译，不做数据同步。也就是说如果用户在 SQLite 下积累了数据然后切到 MySQL，数据会"丢失"（其实还在 SQLite 文件里，但系统不再读取了）。这对用户来说是个坑。短期可以在切换页面上加个数据迁移工具（导出 → 导入），长期应该用统一的迁移框架来管理。

另外 `app.py` 的 `migrate_db()` 函数（L193-L207）用 try/except 做 ALTER TABLE，如果表结构变更比较频繁（后面加技能系统、知识库等都需要改表），这种迁移方式会越来越难维护。

### 8.4 团队能力与资源

坦率地说，我们团队的开发经验和人力都有限。像 RAG、工作流编排、Function Calling 这些功能，技术复杂度都比较高，可能需要较长的学习和调试周期。建议优先做那些技术栈熟悉、参考资料多的功能（安全加固、N+1 修复、流式响应），把复杂的（RAG、工作流）放到后面，边做边学。

### 8.5 外部依赖风险

向量数据库（ChromaDB 或 FAISS）的稳定性和性能在大规模数据下的表现需要验证。如果文档量很大（几千篇以上），可能需要考虑更成熟的向量数据库方案（Milvus、Qdrant）。

采集功能的稳定性依赖于百度新闻、搜狗新闻等网站的页面结构。一旦目标网站改版（HTML 结构变化），BeautifulSoup 的解析逻辑就会失效，需要更新选择器。这是一个持续的维护成本。引入 RSS 源可以部分缓解这个问题——RSS 格式是标准化的，不容易因为网站改版而失效。

OpenAI SDK 的 API 兼容性总体还不错，但如果 DeepSeek 等国内模型在某些细节上和 OpenAI 规范不一致（比如 Function Calling 的支持程度），可能需要在调用层做适配。
