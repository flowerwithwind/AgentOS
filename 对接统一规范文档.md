## 一、接口设计规范

### 1.1 API 路由规范

项目基于 Tornado 框架，所有路由在 `app.py` 的 `make_app()` 函数中统一注册。

#### 路由命名规则

| 类别 | 格式 | 示例 |
|------|------|------|
| 管理侧页面 | `/admin/<模块名>` | `/admin/users` |
| 管理侧API | `/admin/<模块名>/api` | `/admin/users/api` |
| 管理侧操作API | `/admin/<模块名>/api/<动作>` | `/admin/users/api/add` |
| 用户侧页面 | `/user/<模块名>` | `/user/chat` |
| 用户侧API | `/user/<模块名>/api` | `/user/chat/api` |
| 认证相关 | `/auth/<动作>` | `/auth/login` |

#### 路由注册示例（在 `app.py` 中）

```python
# 管理侧
(r"/admin/users", admin.AdminUserListHandler),
(r"/admin/users/api", admin.AdminUserListApiHandler),

# 用户侧（新模块）
(r"/user/chat", user_chat.ChatHandler),
(r"/user/chat/api", user_chat.ChatApiHandler),
```

### 1.2 API 请求与响应规范

#### 请求格式

- **GET 请求**：查询参数通过 URL query string 传递
- **POST 请求**：JSON 格式，`Content-Type: application/json`
- **认证**：通过 Cookie（Tornado `set_secure_cookie`）进行会话认证
- **XSRF 防护**：POST/PUT/DELETE 请求需携带 `X-Xsrftoken` 请求头

```javascript
// 前端 XSRF Token 全局配置示例
$.ajaxSetup({
    beforeSend: function(xhr, settings) {
        if (!/^(GET|HEAD|OPTIONS|TRACE)$/i.test(settings.type)) {
            xhr.setRequestHeader("X-Xsrftoken", document.cookie.match(/\\b_xsrf=([^;]*)/)?.[1] || "");
        }
    }
});
```

#### 响应格式

所有 API 响应统一使用 JSON 格式：

```json
// 成功响应
{
    "code": 0,
    "message": "success",
    "data": { ... }
}

// 分页列表响应
{
    "code": 0,
    "message": "success",
    "data": {
        "items": [ ... ],
        "total": 100,
        "page": 1,
        "page_size": 20
    }
}

// 错误响应
{
    "code": 1,
    "message": "错误描述信息",
    "data": null
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 0=成功，1=业务错误，-1=未登录，-2=无权限 |
| `message` | string | 提示信息 |
| `data` | object/array/null | 返回数据 |

## 二、代码风格要求

### 2.1 Python 后端规范

#### 命名规范

| 类别 | 规范 | 示例 |
|------|------|------|
| 文件名 | `snake_case` | `admin.py`, `user_repository.py` |
| 类名 | `PascalCase` | `UserRepository`, `LoginHandler` |
| 函数/方法 | `snake_case` | `get_user_by_username()` |
| 变量 | `snake_case` | `password_hash`, `url_template` |
| 常量 | `UPPER_SNAKE_CASE` | `DEFAULT_PAGE_SIZE = 20` |
| 私有成员 | `_前置` | `_validate_password()` |
| 模块导入顺序 | 标准库 → 第三方库 → 自定义库 | — |

#### 缩进

- Python 代码统一使用 **制表符（Tab）** 缩进（与现有代码一致）
- 每级缩进宽度 = 4 个空格宽度（即一个 Tab）
- 行尾不留空格

#### 注释规范

```python
# 文件头注释
"""用户管理模块 — 用户数据的增删改查操作"""

# 类注释（简要说明类的职责）
class UserRepository:
    """用户数据访问仓库，封装用户相关的数据库操作"""

# 方法注释
@staticmethod
def get_user_by_username(username):
    """根据用户名获取用户信息
    
    Args:
        username: 用户名
        
    Returns:
        user 字典或 None
    """
```

#### 代码结构规范

```
1. 模块导入（标准库 → 第三方库 → 本项目模块）
2. 常量定义
3. Handler 类（按功能模块组织）
4. Repository 类（每个实体一个）
5. 辅助函数
```

### 2.2 前端规范

#### 模板文件

- 文件名：`snake_case`（如 `admin_user_list.html`）
- 继承 `base.html` 或 `admin.html` 基础模板
- HTML 缩进使用 **2 空格**
- CSS class 命名使用 `-` 连接（如 `.model-card`, `.btn-primary`）

```html
{% extends "admin.html" %}

{% block content %}
<div class="xxx-container">
    <div class="xxx-header">
        <h2>模块标题</h2>
    </div>
</div>
{% end %}

{% block js %}
<script>
// JavaScript 代码
</script>
{% end %}
```

#### JavaScript 规范

- 变量命名：`camelCase`
- 使用 `let` / `const`，不使用 `var`
- LayUI 模块化写法：`layui.use(['table', 'form'], function(){ ... })`
- AJAX 请求统一使用 jQuery 或 LayUI 内置方法

```javascript
// 统一 AJAX 请求示例
function apiRequest(url, method, data, successCallback) {
    $.ajax({
        url: url,
        type: method,
        contentType: 'application/json',
        data: method === 'GET' ? data : JSON.stringify(data),
        success: function(res) {
            if (res.code === 0) {
                successCallback(res.data);
            } else {
                Toast(res.message, 'error');
            }
        },
        error: function() {
            Toast('网络请求失败', 'error');
        }
    });
}
```

## 四、模块对接流程

### 4.1 对接前准备

1. **确认接口定义**：调用方与提供方确认接口名称、参数、返回值
2. **编写接口文档**：在 `docs/` 目录下维护接口文档
3. **Mock 数据准备**：提供方未完成时，调用方可先使用 Mock 数据

### 4.2 对接步骤

```
Step 1: 提供方完成接口开发并自测
    ↓
Step 2: 提供方将代码合并到 develop 分支
    ↓
Step 3: 调用方拉取最新代码（git pull）
    ↓
Step 4: 调用方编写调用代码并测试
    ↓
Step 5: 双方联调，验证数据流是否正确
    ↓
Step 6: 联调通过，关闭对接任务
```

### 4.3 对接问题处理

| 问题类型 | 处理方式 |
|----------|----------|
| 接口返回数据格式不符 | 提供方修改接口，双方确认后重新联调 |
| 接口缺失必要功能 | 提供方补充功能，评估影响范围 |
| 性能不达标 | 双方协商优化方案，必要时调整架构 |
| 依赖其他模块 | 组长协调开发顺序，安排合理的依赖顺序 |

### 4.4 模块依赖顺序

```
第一梯队（基础模块）:
  认证模块 ← 用户管理 ← 功能/角色/权限管理 ← 模型引擎

第二梯队（业务模块）:
  瞭望管理 → 深度采集
  数字员工 → 技能管理
  模型引擎 → 对话功能 → 会话/对话管理

第三梯队（用户侧）:
  对话功能 + 问数功能 + @数字员工 + PDF导出 + 注册 + 文件管理

第四梯队（高级功能）:
  数智大屏 + 智能舆情 + 多数据库支持
```

## 五、版本控制规范

### 5.1 分支命名

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 主分支 | `master` / `main` | `main` |
| 开发分支 | `develop` | `develop` |
| 功能分支 | `feature/<模块名>-<开发者>` | `feature/chat-li` |
| Bug修复分支 | `fix/<问题简述>` | `fix/login-error` |
| 发布分支 | `release/v<版本号>` | `release/v1.0` |

### 5.2 Commit Message 规范

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>
```

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug修复 |
| `docs` | 文档更新 |
| `style` | 代码格式调整（不影响功能） |
| `refactor` | 重构（既不是新功能也不是修复Bug） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖相关 |

**示例：**

```
feat(chat): 实现用户侧对话功能

- 完成消息发送与接收
- 支持对话历史记录查看
- 集成模型引擎API调用
```
